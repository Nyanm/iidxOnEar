# IIDX 键音格式 & 完整曲目重建 调查报告

> 结论先行：**游戏文件里没有现成的完整混音**。我们之前取的"最大块"只是键音库里的 1 号样本（背景底座），
> 难怪听起来是空的。要得到完整歌曲，必须像 BMS 播放器那样，按谱面把成百上千个键音碎片混到一条时间线上。

## 1. 问题本质

IIDX 是键音（keysound）演奏游戏：歌曲不是一段完整音频，而是
**一个键音库（很多短音效）+ 一份谱面（规定每个音效在什么时刻播放）**，运行时实时拼出来。

- 之前 `unpack` 取的 `.s3p`/`.2dx` 最大块 = **键音库的 1 号样本 = 背景底座**（仅环境/底鼓之类，无旋律/键音）。
- 已用 30000 实测确认：谱面在 `t=0` 触发一次 1 号样本，正是那段空底座。

## 2. 文件角色（实测 30000）

| 文件 | 角色 | 细节 |
|------|------|------|
| `.s3p` / `.2dx` | **键音库** | 每个 S3V0/2DX9 块 = 一个短样本。30000.s3p = **1186** 个样本。样本号 N(1基) → 库内第 N-1 项 |
| `.1` | **谱面** | 各难度的事件流，按时刻触发键音 |
| `_pre.2dx` | **预览** | KONAMI 预渲染的选曲试听，单块 **~10s 正确混音**（见第 7 节选项 A） |

样本编码：`.s3p` 的样本是 **WMA(ASF)**；`.2dx` 的样本是 **WAV(MS-ADPCM)**。这点决定了解码难度（第 5 节）。

## 3. 谱面 `.1` 格式（逆向自真实文件）

```
头部 (0x00 起, 14 个槽 × 8 字节):  (u32 offset, u32 length)
  非零槽 = 某难度的事件流。30000 实测: 槽 0/1/2 = SP 三个难度, 槽 6/7/8 = DP 三个难度
  (确切的"槽号→难度名"映射对重建不重要, 见第 4 节)。
事件 (每条 8 字节):  (u32 time_ms, u8 type, u8 param, u16 value)
```

事件类型（实测 30000 SPB，4423 条事件，时长 114.65s）：

| type | 含义 | 计数 | value |
|------|------|------|-------|
| 7 | **自动演奏键音**（到点就播 value 号样本） | 2369 | 1..1172 |
| 2 | **给 lane(param) 指派键音** value（供下一个该 lane 的可见 note 用） | 890 | 2..1186 |
| 0 | **P1 可见 note**（在 lane(param) 上，播该 lane 当前指派的样本） | 1081 | 多为 0(非样本号) |
| 8 | 开场初始 lane→样本 指派 | 6 | — |
| 4 / 5 / 6 | 小节线 / BPM / 结束(time=0x7FFFFFFF) | 各 1 | — |
| 12 / 16 | 杂项/元数据(音符数等) | 72 / 2 | — |

- `value` = **1 基键音号**，直接索引键音库（样本 1 → 库内第 0 项 = 最大的那个底座）。
- 该谱面用到了全部 **1186/1186** 个键音。歌曲时长 = 最后一条真实事件的 time。

## 4. 完整曲目重建算法（BMS 式渲染）

取任意一个难度（**自动演奏 + 该难度可见 note 的样本 = 完整歌曲**；某声音在易难度是自动演奏、在难难度是可见 note，
总和不变，故任选其一即可重建同一首歌）：

1. 把谱面引用到的每个键音解码成 PCM。
2. 开一条 `时长` 的静音立体声 PCM 时间线。
3. 遍历所有"发声"事件——type 7 自动演奏，以及 type 0/1 可见 note（取其 lane 由 type 2/8 当前指派的样本）——
   把该键音的 PCM **叠加**到 `time_ms` 偏移处。
4. 混音（求和 + 限幅/留头），整首一次性编码为 Opus。

需处理的细节：键音可能采样率不一（需重采样到统一率）、单/立体声混合、长键(CN) 的处理、可能的逐事件音量。

## 5. 难点：键音解码

| 曲目族 | 键音编码 | 能否纯 Rust 解码 |
|--------|----------|------------------|
| `.2dx`（v1–24、omni 老歌） | WAV / **MS-ADPCM** | **能**（自写 ADPCM 解码器约百行）→ 可在进程内全程渲染，无额外依赖 |
| `.s3p`（v25+） | **WMA (ASF)** | **不能**（无纯 Rust WMA 解码器；symphonia 也不支持 WMA） |

`.s3p` 的 WMA 解码方案：
- **libavcodec 进程内**（`ffmpeg-next` crate）：把每个 WMA blob 解成 PCM。可行但要链接 ffmpeg 开发库，Windows 上配置较麻烦。
- ❌ 每个键音起一个 ffmpeg 子进程：约 1186/首 × 2373 首 = 数百万次 spawn，太慢，不可行。

## 6. 工作量评估

- 谱面解析：**小**（格式已明）。
- 渲染/混音器：**中**（时间线、叠加、限幅、重采样、长键）。
- s3p 的 WMA 解码：**大**（libavcodec 集成是主要门槛）。
- 总体：相当于写一个"小型 IIDX 音频渲染器"，有界但不轻——与用户判断一致。

## 7. 选项与建议

- **选项 A — 用预览片段（简单、正确、但短）**：`_pre.2dx` 是 KONAMI 预渲染的选曲试听，单块 ~10s 的**正确混音**。
  把提取目标从"最大块"改成"预览文件整体"即可，**几乎零新代码**（我们本就能解 `.2dx`）。代价：每首只有 ~10s。
  → **建议先听一个 `_pre.2dx` 确认它确实是完整混音**（我判断是，但你的耳朵说了算）。
- **选项 B — 完整键音渲染（真·完整曲）**：第 4 节的渲染器。工作量大，WMA 解码（libavcodec）是主要门槛。
- **选项 C — 混合**：老歌（`.2dx`，纯 Rust ADPCM，无新依赖）走完整渲染；新歌（`.s3p`）暂用预览，或上 libavcodec。

**推荐路径**：先听预览定调——若 ~10s 短片可接受，走 A（基本免费）；若必须完整曲，承认这是个大工程，
建议从 C 起步（先把纯 Rust 的 `.2dx` 完整渲染跑通验证算法，再啃 `.s3p` 的 WMA 解码）。
